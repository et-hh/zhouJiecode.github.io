# 对比

eslint用于代码质量的检测，例如vue/no-use-v-if-with-v-for这个规则用来约束vue代码中v-for和v-if不能用于同一元素，其中存在效率问题，详见。eslint同样也可用于代码样式的检测，不过配置较为复杂，且对样式控制的颗粒度不够细

prettier专注于统一代码样式并格式化代码，代码风格美观，配置简单，已逐渐成为业界主流的代码风格格式化工具。因此，将它与eslint结合，可减轻eslint的校验规则，使eslint专注于检测代码质量，而prettier专注于代码样式，分工明确

# 配置（vue项目）

# 1. vscode中安装vetur、eslint、prettier插件

vetur插件主要用于.vue文件的高亮显示，其也会对vue的一些规则做代码检测，如上面提过的no-use-v-if-with-v-for规则，其会在编写代码时实时显示在IDE上。eslint和prettier插件主要用于IDE的实时检测

# 2. 安装所需的npm包

  • eslint和prettier，可全局安装或者项目内安装，安装的目的是IDE运行插件所需
  • eslint-loader，主要用于webpack构建时进行eslint检测
  • eslint-plugin-vue，使得eslint能够检测.vue文件并能按照vue代码风格指南的规则检测代码质量，其与eslint-plugin-html功能接近且冲突，因此只用前者就够了
  • eslint-plugin-prettier，使得eslint能够处理prettier的规则
  • eslint-config-prettier，为了防止prettier的规则与eslint有冲突，禁用了eslint所有关于代码样式的规则
  • husky，用于提供git commit hook
  • lint-staged，能够对本次提交的文件进行eslint检测并fix error，防止将带error的代码直接提交到仓库

# 3. .eslintrc.js配置

关于eslint配置的详细信息这里不做赘述，这里只介绍和搭建vue项目相关的配置，一个简单的配置可参考如下：

module.exports = {
  root: true,
  env: {
    browser: true,
    node: true
  },
  parserOptions: {
    parser: 'babel-eslint'
  },
  extends: ['plugin:vue/essential', 'eslint:recommended', 'plugin:prettier/recommended'],
  // required to lint *.vue files
  plugins: ['vue'],
  // add your custom rules here
  rules: {
    'no-console': process.env.NODE_ENV === 'production' ? 'error' : 'off',
    'no-debugger': process.env.NODE_ENV === 'production' ? 'error' : 'off',
    'vue/max-attributes-per-line': [
      2,
      {
        singleline: 20,
        multiline: {
          max: 1,
          allowFirstLine: false
        }
      }
    ]
  }
}

extends选项是用来继承某些rules用的，'plugin:vue/essential'继承了与vue相关的rules，具体可参考这里。'plugin:prettier/recommended'干了三件事：

  • 开启了eslint-plugin-prettier
  • 设置名为prettier/prettier的rule为'error'
  • 继承了eslint-config-prettier的相关配置

# 4. vscode中配置eslint

我们需要vscode能在开发时就能实时的检测，因此需要在vscode中安装eslint插件，然后需要在settings.json中写上相应配置，参考如下：

"editor.formatOnSave": true, // 保存时格式化代码
"prettier.eslintIntegration": true,
"eslint.autoFixOnSave": true
"eslint.options": {
    "plugins": ["vue"]
},
"eslint.validate": [
    "javascript",
    "javascriptreact",
    "html",
    "vue"
],

# 5. 添加git commit hook

为了避免团队小伙伴将带有error的代码push到仓库，我们需要在提交代码时再做一层防范，在git commit时自动检测并修复此次提交的文件，配置如下：

"husky": {
    "hooks": {
    "pre-commit": "lint-staged"
    }
},
"lint-staged": {
    "*.{js,vue}": [
        "npm run lint",
        "git add"
    ]
}

# 结合vue cli 3

若是使用vue cli 3搭建的项目，那会出现一种情况，执行npm run lint时有error提示，但是在启动server或是保存文件时，终端都没有error提示。这是因为脚手架内集成了cache-loader，该loader为了避免在webpack编译时一些开销昂贵的操作，将结果缓存在磁盘中，可在node_modules/.cache文件夹中查看到。而我们在配置eslint-loader时都会加上enfoce:pre选项，是为了开启前置调用，防止代码优先被vue-loader等loaders转化。因此如果编译结果被缓存，那么在保存文件执行热更新时由于vue代码已被转化成原生代码，lint将不会生效，因此看不到error提示。查阅vue cli官方文档，可以看到有一个lintOnSave选项， 我们可在vue.config.js中对其进行设置：

module.exports = {
    lintOnSave: process.env.NODE_ENV !== 'production' ? 'error' : false,
}

这样在开发时我们就能在终端实时看到报错信息了，不过为了节省开销，vue cli并没有对全文件进行lint，因此只能看到当前修改文件的lint信息。
